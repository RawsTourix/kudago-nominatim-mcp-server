# Отчёт о live-тестировании Transitous, Geoapify и Google Routes

> [!IMPORTANT]
> **Главный практический вывод:** ни один из трёх проверенных API не подходит
> для надёжного построения маршрутов общественного транспорта по России.
> Transitous не показал транспортных данных для Москвы и Московской области,
> Geoapify строит там только приблизительный маршрут без расписания, а Google
> Routes возвращает `no_route`.

| Параметр | Значение |
|---|---|
| Дата фиксации результатов | 16 июля 2026 года |
| Дата маршрутов, где API поддерживает время | 17 июля 2026 года |
| Основной регион проверки | Москва и Московская область |
| Основные маршруты | Нахабино → Курская; Нахабино → Архангельское |
| Международные контроли | Берлин, Нью-Йорк, Токио, Найроби, Хараре |
| Проверенные API | Transitous / MOTIS 2, Geoapify Routing, Google Routes API |
| Решение для России | **Не использовать ни один как основной transit-provider** |

## Краткий вердикт

```text
Нужен подтверждённый маршрут общественного транспорта по России
│
├─ Transitous ─── нет наблюдаемого покрытия Москвы и области
├─ Geoapify ───── есть только приблизительная модель без расписания
└─ Google Routes  возвращает HTTP 200, но routes=[]
                         │
                         └─ ни один провайдер не выполняет контракт
```

Обозначения в таблицах:

- ✅ — требование подтверждено live-ответом;
- ⚠️ — частичный или приблизительный результат, который нельзя выдавать за
  подтверждённый маршрут;
- ❌ — требование не выполнено;
- `no_route` — запрос принят провайдером, но маршрут не найден;
- `provider_error` — провайдер вернул ошибку вместо маршрута.

### Сводная оценка

| Критерий | Transitous | Geoapify | Google Routes |
|---|---|---|---|
| Нахабино → Курская | ❌ 0 маршрутов | ⚠️ только `approximated_transit` | ❌ `no_route` |
| Нахабино → Архангельское | ❌ 0 маршрутов | ⚠️ только `approximated_transit` | ❌ `no_route` |
| Остановки и линии Москвы | ❌ не обнаружены | ❌ не возвращены | ❌ не возвращены |
| МЦД-2 | ❌ не обнаружен | ❌ структурно не распознан | ❌ не обнаружен |
| Расписание отправления и прибытия | ❌ нет данных | ❌ API не даёт расписание | ❌ нет маршрута для России |
| Планирование на заданное время | ⚠️ API принимает время, но данных нет | ❌ временных параметров нет | ⚠️ API принимает время, но маршрут не найден |
| Успешный международный контроль | ✅ Берлин | ⚠️ Берлин и Нью-Йорк без расписания | ✅ Берлин, Нью-Йорк и Найроби |
| Допустимая роль в проекте | Только отдельные покрытые регионы | Только явно приблизительный fallback | Только отдельные покрытые регионы |
| Пригодность для России | **❌ нет** | **❌ нет** | **❌ нет** |

Google Routes оказался самым содержательным из трёх в покрытых регионах:
он возвращал остановки, линии и времена. Однако для целевого российского
сценария это не помогает — все московские варианты остались пустыми.

## Что означало «подходит для России»

Провайдер считался пригодным только в том случае, если он мог одновременно:

1. построить маршрут Нахабино → Курская и маршрут Нахабино → Архангельское;
2. вернуть реальные участки общественного транспорта, а не только геометрию;
3. назвать остановки и линии;
4. вернуть времена отправления и прибытия, связанные с выбранной датой;
5. показать железнодорожный участок МЦД-2 либо его корректный эквивалент;
6. стабильно повторить результат;
7. пройти международный контроль, подтверждающий исправность запроса и
   парсера.

Ни один API не выполнил этот набор требований для Москвы и Московской области.

## Российские сценарии в одном представлении

| Провайдер | Нахабино → Курская | Нахабино → Архангельское | Доверие к результату |
|---|---|---|---|
| Transitous | HTTP 200, 0 itineraries | HTTP 200, 0 itineraries | Покрытие отсутствует |
| Geoapify `transit` | `provider_error` | `provider_error` | Маршрут не подтверждён |
| Geoapify `approximated_transit` | 35 586 м, 2 519 с | 16 119 м, 2 542 с | Только приблизительная модель |
| Google Routes | HTTP 200, `routes=[]` | HTTP 200, `routes=[]` | `no_route` |

Критически важно: геометрия Geoapify `approximated_transit` не эквивалентна
маршруту по расписанию. Она не подтверждает конкретный поезд, станцию,
пересадку, время отправления или доступность рейса в выбранную дату.

## 1. Transitous / MOTIS 2

### Цель диагностики

Проверка должна была отделить три возможные причины пустого ответа:

- ошибку MCP-интеграции;
- проблему привязки координат к остановкам или слишком строгий пеший доступ;
- отсутствие транспортных данных у самого провайдера.

Для этого использовались официальные endpoints MOTIS:

- `/api/v6/map/stops` — остановки в области;
- `/api/v6/stoptimes` — отправления и прибытия ближайшей остановки;
- `/api/v6/plan` — построение маршрута.

### Наблюдаемые остановки и расписания

| Область | Остановки | Stoptimes | Результат |
|---|---:|---:|---|
| Berlin Alexanderplatz | 108 | 8 | ✅ данные есть |
| Berlin Hauptbahnhof | 84 | 6 | ✅ данные есть |
| Станция Нахабино | 0 | 0 | ❌ данных нет |
| Метро Курская | 0 | 0 | ❌ данных нет |
| Центр Москвы | 0 | 0 | ❌ данных нет |
| Архангельское | 0 | 0 | ❌ данных нет |

Нулевые `stoptimes` у российских точек являются следствием отсутствия
остановок: для них не нашлось stop ID, по которому можно запросить расписание.

### Матрица маршрутизации

Были проверены содержательные варианты, а не полный декартов перебор:

- время отправления: 08:00, 12:00 и 18:00;
- `TRANSIT` и явный набор `SUBURBAN,SUBWAY,BUS`;
- координаты и stop ID там, где они существовали;
- пеший доступ 900 и 1800 секунд;
- `radius=500` и `radius=1000` для независимой проверки matching;
- Нахабино → Курская, центр Москвы → центр Москвы и
  Нахабино → Архангельское;
- Берлин coordinate-to-coordinate и stop-to-stop как контроль.

Результат:

- Берлин: 3 маршрута coordinate-to-coordinate и 3 маршрута stop-to-stop;
- все российские варианты: HTTP 200 и 0 маршрутов;
- увеличение пешего доступа и `radius` не изменило результат;
- `debugOutput` для российских запросов показывал
  `n_start_offsets=0`, `n_dest_offsets=0`, `n_routes_visited=0`.

То есть движок не дошёл до выбора маршрута: рядом с началом и назначением не
было доступных транспортных данных.

### Проверка каталога источников

Transitous собирает публичные GTFS-источники. На дату проверки в официальном
[каталоге Transitous](https://transitous.org/sources/) не были найдены:

- `Russia`;
- `Moscow` или `Moskva`;
- `Nakhabino`;
- `RZD`;
- `Mosgortrans`.

Отсутствие текста в каталоге само по себе ещё не было бы достаточным
доказательством. Здесь оно совпало с тремя независимыми наблюдениями: нет
остановок, нет расписаний и нет маршрутов, тогда как контрольный Берлин
работает.

### Ресурсный бюджет и результат

| Показатель | Значение |
|---|---:|
| Максимальный бюджет диагностики | 29 запросов |
| Фактически выполнено | 23 запроса |
| Unit-тесты диагностики | 10 passed |
| Полный pytest на том этапе | 177 passed, 2 skipped |

Итоговая классификация:

```text
provider_coverage_unavailable
```

> [!CAUTION]
> Transitous нельзя использовать для маршрутизации по Москве и Московской
> области в текущем виде. Повторная настройка walking access, времени или
> радиуса не компенсирует отсутствие исходных transit-данных.

## 2. Geoapify Routing API

### Что сравнивалось

Для каждой точки сравнивались два разных режима:

- `transit` — обычный режим общественного транспорта;
- `approximated_transit` — приблизительный маршрут, который, согласно
  [документации Geoapify](https://apidocs.geoapify.com/docs/routing/),
  строится по доступным в OpenStreetMap transit routes.

Проверены два российских маршрута и пять международных контролей. Для
Нахабино → Курская и Берлина дополнительно выполнены повторные запросы.

### Полная live-матрица

| Сценарий | Режим | Результат | Дистанция | Длительность |
|---|---|---|---:|---:|
| Нахабино → Курская | `transit` | `provider_error` | — | — |
| Нахабино → Курская | `approximated_transit` | приблизительный маршрут | 35 586 м | 2 519 с |
| Нахабино → Архангельское | `transit` | `provider_error` | — | — |
| Нахабино → Архангельское | `approximated_transit` | приблизительный маршрут | 16 119 м | 2 542 с |
| Берлин | `transit` | структурированный, без расписания | 3 409 м | 754 с |
| Берлин | `approximated_transit` | приблизительный маршрут | 3 320 м | 479 с |
| Нью-Йорк | `transit` | структурированный, без расписания | 1 169 м | 683 с |
| Нью-Йорк | `approximated_transit` | приблизительный маршрут | 1 133 м | 490 с |
| Токио | `transit` | `provider_error` | — | — |
| Токио | `approximated_transit` | `provider_error` | — | — |
| Найроби | `transit` | только generic route | 1 647 м | 1 446 с |
| Найроби | `approximated_transit` | `provider_error` | — | — |
| Хараре | `transit` | `provider_error` | — | — |
| Хараре | `approximated_transit` | `provider_error` | — | — |

Для московского `transit` провайдер вернул внутренний `statusCode=400`:

```text
Cannot reach destination - too far from a transit stop
```

### Почему приблизительный маршрут недостаточен

В российских ответах `approximated_transit`:

- не распознана МЦД-2;
- не возвращены названия линий, маршрутов и остановок;
- нет времени отправления и прибытия;
- нет service date, расписания и realtime-признаков;
- нельзя передать требуемую дату или время отправления;
- поле `time` означает расчётную длительность в секундах, а не время по
  расписанию.

Типы инструкций с названием `Transit` показывают только характер
приблизительной геометрии. Они не подтверждают существование конкретного
транспортного рейса.

### Стабильность и ресурсный бюджет

- выполнено 16 API-запросов: 14 основных и 2 повтора;
- повторы Нахабино → Курская и Берлина совпали после исключения geometry и
  технической metadata;
- временные raw-файлы были удалены;
- API key не выводился и не сохранялся;
- unit-тесты диагностики: `14 passed`;
- полный pytest на том этапе: `191 passed, 2 skipped`.

Итоговая классификация:

```text
recommended_only_as_approximate_fallback
```

> [!WARNING]
> Эта классификация не означает, что Geoapify подходит как transit-provider.
> Режим можно рассматривать только как отдельно названный приблизительный
> fallback с низким уровнем доверия. Он не должен обслуживать контракт
> маршрута по расписанию.

## 3. Google Routes API

### Что проверялось

Запросы использовали `travelMode=TRANSIT`, явный field mask с
`transitDetails`, временем, остановками и линиями. Матрица включала:

- семь базовых региональных сценариев;
- `arrivalTime` для Москвы и Берлина;
- `FEWER_TRANSFERS` для Нахабино → Курская;
- повтор Москвы и повтор Берлина.

|---|---:|---:|---|---:|---:|---:|---:|---|

`Times` в таблице — число transit-участков, для которых одновременно получены
структурированные времена отправления и прибытия.

### Что показал контроль

Google Routes подтвердил, что запрос, field mask и parser работают:

- в Берлине, Нью-Йорке и Найроби получены остановки, линии и расписание;
- arrive-by работает в Берлине;
- в Берлине и Нью-Йорке получена информация о стоимости;
- повторные запросы Москвы и Берлина совпали после исключения технической
  metadata;
- provider errors не наблюдались.

При этом для российских запросов:

- не подтверждены Нахабино и Курская как transit stops;
- не найден МЦД-2 или эквивалентный железнодорожный участок;
- нет остановок, линий, расписания и fare;
- arrive-by и `FEWER_TRANSFERS` не меняют `no_route`.

Успешные международные ответы содержали один основной маршрут и пять
альтернатив. Это фактически наблюдаемый ответ API; он сохранён как факт, даже
если общий REST reference описывает другое максимальное количество.

### Ресурсный бюджет и результат

- основная матрица: 12 запросов;
- дополнительно: 1 контрольный запрос после исправления parser;
- всего в ходе итоговой диагностики: 13 запросов;
- raw responses и API key на диск не записывались;
- unit-тесты диагностики: `22 passed`;
- полный pytest на том этапе: `213 passed, 2 skipped`.

Итоговая классификация:

```text
recommended_only_as_regional_provider
```

> [!CAUTION]
> Google Routes можно рассматривать для отдельно подтверждённых регионов, но
> нельзя использовать как основной провайдер проекта, ориентированного на
> Россию. Сильные ответы в Германии, США и Кении не компенсируют `no_route` в
> Москве и Московской области.

## Почему это не похоже на ошибку нашей интеграции

### 1. Контрольные регионы работают

| Провайдер | Положительный контроль |
|---|---|
| Transitous | Берлин: остановки, stoptimes и маршруты |
| Geoapify | Берлин и Нью-Йорк: структурированный transit-ответ без расписания |
| Google Routes | Берлин, Нью-Йорк и Найроби: полноценные transit details |

Один и тот же diagnostic layer способен отправить запрос, получить ответ и
нормализовать данные. Ошибка общего HTTP-клиента или parser сломала бы и
контроли.

### 2. Координаты проверялись последовательно

Для всех трёх API использовались одни и те же смысловые опорные точки.
Координаты Нахабино и Курской совпадали точно; для Архангельского использовались
две близкие точки внутри одной целевой области:

| Точка | Широта | Долгота |
|---|---:|---:|
| Станция Нахабино | 55.8415879 | 37.1849110 |
| Метро Курская | 55.7588462 | 37.6580446 |
| Центр Москвы | 55.7512440 | 37.6184230 |
| Архангельское | около 55.79 | около 37.29 |

Transitous дополнительно проверял остановки вокруг каждой точки и не нашёл их.
Это исключает объяснение, при котором маршрута нет только из-за центра
населённого пункта вместо станции.

### 3. Изменение параметров не восстановило Россию

- Transitous: не помогли другое время, явные modes, увеличенный walking access
  и radius;
- Geoapify: обычный `transit` продолжил ошибаться, а approximate-режим остался
  без расписания;
- Google Routes: не помогли arrive-by и `FEWER_TRANSFERS`.

### 4. Провайдеры проявили разные, но согласованные симптомы

| Провайдер | Наблюдаемый симптом | Интерпретация |
|---|---|---|
| Transitous | нет stops, stoptimes и itineraries | нет исходного покрытия |
| Geoapify | обычный transit не достигает остановки; approximate строится | есть только приближённая OSM-модель |
| Google Routes | корректный HTTP 200 с пустым `routes` | маршрут в данных провайдера не найден |

Разные API не повторяют одну и ту же техническую ошибку. Они независимо
показывают, что требуемого российского timetable-покрытия нет.

## Архитектурное решение

| Провайдер | Решение для production |
|---|---|
| Transitous | Не использовать для России; оставить только для отдельно проверенных регионов, если они понадобятся |
| Geoapify | Не подключать к timetable-контракту; максимум — отдельная функция приблизительной геометрии с явной маркировкой |
| Google Routes | Не использовать как основной российский provider; возможен отдельный региональный adapter |

Для российского `plan_public_transport` нужен источник, который реально
содержит российские остановки, линии и расписания. Практические направления:

1. проверить провайдера, ориентированного на российский транспорт;
2. проверить доступность официальных GTFS или иных машиночитаемых feeds у
   железнодорожных и городских операторов;
3. при нескольких источниках построить собственный слой агрегации и
   нормализации;
4. возвращать честное `coverage_unavailable`, если подтверждённого расписания
   нет;
5. никогда не преобразовывать approximate geometry в «подтверждённый» маршрут
   с выдуманными остановками или временем.

> [!NOTE]
> Fallback-цепочка из этих трёх API не решает проблему. Transitous и Google не
> дают маршрут для целевого региона, а Geoapify не даёт расписание. Переключение
> между ними меняет форму отказа, но не создаёт недостающие транспортные данные.

## Ограничения вывода

Live-тест — снимок состояния провайдеров, а не вечная гарантия их покрытия.
Вывод ограничен следующими условиями:

- проверка зафиксирована 16 июля 2026 года;
- российская часть сфокусирована на Москве и Московской области;
- детально проверены два целевых маршрута и четыре области остановок;
- данные и контракты внешних API могут измениться;
- тест не доказывает математически отсутствие любого маршрута в каждой точке
  России.

Тем не менее этого достаточно для продуктового решения: основной сценарий
проекта находится именно в Москве и Московской области, и все три провайдера
провалили обязательные маршруты или обязательное качество результата.

## Воспроизведение

### Переменные окружения

Секреты задаются только локально в `.env` и не должны попадать в отчёт или git.

```dotenv
TRANSITOUS_USER_AGENT=application/version contact@example.com
GEOAPIFY_API_KEY=<secret>
GOOGLE_MAPS_BASE_URL=https://routes.googleapis.com/directions/v2:computeRoutes
GOOGLE_MAPS_API_KEY=<secret>
```

### Live-скрипты

```powershell
python scripts/test_transitous_coverage_live.py --date 2026-07-17
python scripts/test_geoapify_transit_live.py
python scripts/test_google_routes_transit_live.py
```

Transitous сохраняет детальный JSON в системную временную директорию, если не
передан `--output`. Geoapify удаляет временные raw-файлы после формирования
сводки. Google Routes не сохраняет raw responses на диск.

### Unit-тесты диагностик

```powershell
python -m pytest -q tests/test_transitous_coverage_live.py
python -m pytest -q tests/test_geoapify_transit_live.py
python -m pytest -q tests/test_google_routes_transit_live.py
```

Числа полного pytest выше относятся к последовательным снимкам репозитория:
после Transitous, после Geoapify и после Google Routes соответственно. Поэтому
они увеличиваются от раздела к разделу.

## Финальный вывод

> **Transitous, Geoapify и Google Routes не подходят для российского
> public-transport routing в требуемом качестве.**

- Transitous: для проверенных московских точек отсутствуют даже stops и
  stoptimes;
- Geoapify: доступна только приблизительная геометрия без расписания и
  структурированного МЦД-2;
- Google Routes: полноценен в части зарубежных регионов, но для всех
  российских вариантов возвращает `no_route`.

Следующий провайдер должен оцениваться прежде всего по наблюдаемому покрытию
России и полноте расписаний, а не только по наличию формального transit-режима
в API.
